K-Sutda — 디자인 거버넌스 (Design Governance)

디자인 자산 변경 관리 / 버전 관리 / PR 워크플로우 / 일관성 검증. 디자이너 ↔ 개발자 ↔ 외부 협업자 (작곡가 등) 의 협업 규칙.

0. 거버넌스 필요성

0-1. 왜 거버넌스인가

K-Sutda 는:

  • 솔로 개발자 + 외부 디자이너 / 작곡가 협업 — 책임·권한 명확 필요
  • 15 자료 × .md+.html = 32 파일 — 변경 추적 어려움
  • 베타 → 정식 출시 → 업데이트 — 디자인 일관성 유지 필수
  • 글로벌 출시 — 다국어 / 다플랫폼 자산 버전 관리

거버넌스 없으면:

  • 같은 컴포넌트 다른 시각 (디자인 분기)
  • 카피 톤 일관성 깨짐 ("다이" vs "포기" 혼재)
  • 자산 라이선스 추적 실패
  • 변경 사항 알 수 없음 (디자이너 ↔ 개발자 소통 어려움)

1. 디자인 자산 분류 / 위치

1-1. 자산 카테고리

카테고리 위치 변경 권한
디자인 자료 (가이드) design-handoff/*.md + .html 개발자 + 디자이너 합의
브랜드 자산 (로고 / 카피) assets/marketing/brand/ 개발자 최종 승인
게임 자산 (카드 / UI / 캐릭터) assets/cards/ assets/ui/ assets/mapa/ 디자이너 작업 → 개발자 통합
음향 자산 assets/audio/ 작곡가 / 사운드 디자이너 → 개발자 통합
코드 (디자인 토큰) scripts/ui/Palette.gd 개발자 (디자이너와 합의 후)
Figma 디자인 외부 — Figma 워크스페이스 디자이너 메인

1-2. 자산 명명 규칙

종류 규칙 예시
가이드 문서 NN-kebab-case.md 01-brief.md
이미지 snake_case.png card_1_gwang.png
폰트 Name-Weight.ttf Pretendard-Bold.otf
사운드 snake_case.ogg card_flip.ogg
Figma 페이지 ## NN. Page Name ## 01. MainMenu

2. 변경 관리 워크플로우

2-1. 자산 변경 단계

[디자이너 변경 의도]
        ↓
[Figma 시안 작성 / 자산 제작]
        ↓
[개발자 리뷰 (이 가이드 위배 X?)]
        ↓
[자산 ./assets/ 통합 + git commit]
        ↓
[design-handoff/*.md 갱신 (필요 시) + .html 재생성]
        ↓
[git PR / push]
        ↓
[테스트 (인게임 시각 확인)]
        ↓
[Discord / Slack 변경 알림]

2-2. 변경 분류

변경 종류 처리
마이너 (텍스트 1줄 / 색 미세 조정) 직접 commit, 가이드 일관
메이저 (컴포넌트 신규 / 컬러 추가) 사전 합의 → 디자인 자료 갱신 → 코드
브랜드 (로고 / 카피) 개발자 + 디자이너 합의 + 마케팅 영향 검토
공개 자산 (스토어 페이지 / 트레일러) 공개 전 최종 합의

2-3. 합의 도구

  • 빠른 합의: Slack / Discord / 카톡 메시지
  • 중요 합의: Figma 코멘트 + 명확한 ✅
  • 공개 자산: 별도 회의 (Zoom / Google Meet)

3. 버전 관리 (Semantic Versioning)

3-1. 디자인 자료 버전

각 가이드 (design-handoff/*.md) 의 헤더에 버전:

# K-Sutda — 디자이너 브리프

> Version: 1.2.0
> Last Updated: 2026-05-16
> 작업 시작 전 반드시 ...

3-2. SemVer 규칙

  • MAJOR (1.x.x → 2.0.0): 게임 디자인 큰 변경 (예: 2단계 베팅 도입)
  • MINOR (x.1.x → x.2.0): 새 컴포넌트 / 캐릭터 / 화면 추가
  • PATCH (x.x.1 → x.x.2): 텍스트 수정 / 사양 미세 조정

3-3. 게임 자체 버전

project.godot 의 application/config/version 와 동기:

  • v0.x.x — 베타 (현재)
  • v1.0.0 — 정식 출시
  • v1.x.x — 정식 출시 후 패치

3-4. CHANGELOG (옵션)

design-handoff/CHANGELOG.md 신규 (선택):

### Added

4. Git 워크플로우

4-1. 브랜치 전략

main                ← 안정 / 출시 가능 상태
├── feat/sprint-3.x ← 개발 중 sprint
├── design/figma-v2 ← 디자이너 작업 브랜치 (옵션)
└── docs/handoff-update ← 디자인 자료 갱신

4-2. Commit Message 규칙

[type]([scope]): [요약 한국어]

[상세 설명 — 변경 사유 / 영향]

Co-Authored-By: [협업자] <email>

Type:

  • feat — 새 기능 / 자산
  • fix — 버그 / 오류 수정
  • docs — 가이드 / 문서
  • style — 디자인 시각 변경
  • refactor — 코드 리팩토링
  • test — 테스트
  • chore — 빌드 / 설정

Scope:

  • design — design-handoff/ 변경
  • brand — 로고 / 카피
  • assets — 게임 자산
  • audio — 음향
  • code — 게임 코드

예시 (이 repo 의 실제 commit):

docs(design): 12~15 자료 추가 — 모바일/이펙트/문화/브랜드

5. 리뷰 체크리스트

5-1. 디자인 자료 PR 리뷰

새 가이드 / 갱신 시:

  • 마크다운 형식 정확 (헤더 / 표 / 코드블록)
  • .html 동시 생성 (scripts/tools/md_to_html.sh)
  • 다른 가이드와 일관 (예: Palette 색 같은 hex 사용)
  • 한국어 / 영어 톤 일관 (15-brand-identity)
  • 마이크로카피 어휘 (08-ui-ux-guidelines)
  • 접근성 고려 (09-accessibility)
  • 디자이너 / 외부 협업자가 5분 안에 이해 가능

5-2. 새 컴포넌트 / 화면 PR 리뷰

  • Palette 색만 사용
  • 8px 그리드 (spacing)
  • 폰트 위계 (Title/Heading/Body/Caption)
  • 4 상태 (default/hover/pressed/disabled)
  • 모션 04-motion-spec 사양 따름
  • 모바일 적응 (12-mobile-ui)
  • 접근성 (명도 / 폰트 / 터치 타겟)
  • localization 준비 (i18n 키)
  • 컴포넌트 라이브러리 (11) 등록

5-3. 캐릭터 / 카드 일러스트 PR 리뷰

  • 한국적 톤 (14-cultural-guide)
  • 광패 / 희귀도별 강조 (05-card-art-style)
  • 해상도 / 형식 (02-asset-spec)
  • UI overlay 안전 영역
  • 라이선스 명시 (work-for-hire / 본인 창작)

5-4. 사운드 PR 리뷰

  • 파일 키 일치 (AudioBus.SOUNDS)
  • 포맷 / 비트레이트 (06-music-direction)
  • 볼륨 균형 (-12 ~ -20 dB)
  • 라이선스 명시

6. 일관성 검증 도구

6-1. 수동 검증

분기별 (3개월) 1회 디자인 일관성 audit:

  • 모든 화면에서 같은 색 = 같은 의미?
  • 같은 컴포넌트 시각 일치?
  • 마이크로카피 톤 일관?
  • 새 자산 라이선스 등록?
  • design-handoff/*.md 와 실제 코드 / 자산 일치?

6-2. 자동 검증 (미래)

  • Figma 토큰 vs Palette.gd: Figma Variables → Palette.gd 동기화 스크립트
  • i18n 키 검증: 모든 tr() 키가 .csv 에 존재
  • 이미지 사이즈 검증: 화투 220×320, 마패 200×280 자동 확인
  • 명도 대비 자동: 새 컬러 조합 PR 시 4.5:1 검증

6-3. 디자인 시스템 사이트 (옵션)

design-handoff/ 의 HTML 들을 정적 사이트로 통합:

  • GitHub Pages 또는 Vercel 자동 배포
  • 디자이너 / 협업자 항상 최신 자료 접근
  • 검색 / 네비게이션 기능

13. 비상 대응

13-1. 디자이너 / 협업자 이탈

  • 모든 자산 git + Drive 백업 (이중)
  • 작업 중인 Figma 파일 정기 export
  • 다른 디자이너에게 인계 가능한 자료 (design-handoff)

13-2. 라이선스 / 저작권 분쟁

  • 모든 외주 work-for-hire 계약서 보관
  • 라이선스 충돌 시 즉시 해당 자산 교체 (대안 준비)
  • 법무 자문 (정식 출시 전 검토)

13-3. 디자인 일관성 회귀 (큰 변경 후)

  • 분기 1회 audit
  • 회귀 발견 시 PR 으로 일괄 수정
  • 빈도 잦으면 자동 검증 도구 도입

14. 결론

좋은 디자인 거버넌스 = 협업자가 자유롭게 작업하되,
브랜드와 일관성은 무너지지 않는 시스템.

K-Sutda 의 거버넌스 핵심:
1. 자산 명명 / 위치 규칙 명확
2. 변경은 PR / commit 로 추적
3. 라이선스 모두 등록
4. 디자인 자료 (design-handoff) 항상 최신
5. 사용자 페르소나 / 브랜드 가이드가 분쟁 해결 기준

거버넌스 = 자유의 토대.
