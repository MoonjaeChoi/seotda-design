1. 시각 회귀 테스트

1-1. 시각 회귀란?

디자인 변경 후 의도된 변경만 일어났는지, 다른 화면이 깨지지 않았는지 검증.

1-2. 수동 체크리스트

새 디자인 변경 후 다음 화면 모두 확인:

  • MainMenu — 타이틀 / 버튼 4~5개 / 닉네임 입력 / 배경
  • Board — 카드 4장 / 베팅 패널 / 명성치 / pot / 마패 슬롯
  • Market — 마패 카드 4장 / 슬롯 / 명성치 / 리롤
  • Leaderboard — 정렬 / 닉네임 강조
  • DailyActivity — 카드 3개 / 14일 차트
  • HelpInfo — RichText 스크롤
  • ProbabilityInfo — RichText 스크롤

각 화면에서:

  • 색 일관 (Palette)
  • 폰트 일관
  • 8px 그리드
  • 모션 부드러움 (60 FPS)

1-3. 자동 시각 회귀 (미래)

  • Screenshot diff — 이전 / 현재 스크린샷 비교
  • Godot 헤드리스 → 화면 캡처 → 픽셀 diff
  • 임계값 (예: 5% 픽셀 다름) 초과 시 알림

→ Phase 4+ 도입 검토.

2. 사용성 테스트 (Usability Testing)

2-1. 5명의 법칙

사용성 테스트는 5명만으로 85% 문제 발견 (Jakob Nielsen). 베타 모집 후 코어 5명 우선 테스트.

2-2. 테스트 프로토콜

[준비]
- 테스트 시간: 1인 30~45분
- 환경: 사용자 평소 환경 (자기 PC / 모바일)
- 도구: 화면 녹화 (사전 동의) + 보이스 (생각 말하기)
- 진행자: 개발자 (직접) 또는 별도 진행자

[순서]
1. 인사 + 동의 (5분)
2. 페르소나 확인 (어떤 사용자인가)
3. 게임 첫 실행 — 사용자 자율 (10~15분)
   - 진행자 안내 ❌
   - 사용자 "생각 말하기" (어디 클릭하지? 왜?)
4. 단계 1 완료까지 진행
5. 인터뷰 (10분)
   - 혼란스러웠던 부분
   - 좋았던 부분
   - 다시 해보고 싶나?
6. 마무리 (5분)

2-3. 측정 지표

지표 측정
시간 1단계 클리어 첫 실행 → 단계 1 완료 (목표 10분 이내)
튜토리얼 이탈 튜토리얼 스킵한 사용자 %
첫 라운드 fold 1차 dia 비율 (페르소나별)
블러프 첫 시도 약한 패로 raise 시도한 라운드 %
재플레이 의지 "다시 해보고 싶다" 응답 %
추천 의지 NPS (Net Promoter Score)

2-4. 페르소나별 테스트 우선순위

페르소나 (16) 테스트 우선순위
P5 캐주얼 🔴 1순위 — 가장 약한 고리
P2 화투 친숙 중년 🟡 2순위 — 모바일 / 한국 정체성
P1 발라트로 코어 🟢 3순위 — 깊이 검증
P3 글로벌 🟢 영문 출시 직전
P4 K-문화 글로벌 🟢 영문 출시 후

5. 디자인 검증 — 디자이너 self-check

5-1. 새 디자인 시안 검증

(08-ui-ux-guidelines §10 + 09-accessibility §12 통합)

  • Palette 색만 사용
  • 폰트 위계 5단계 중 적절
  • 8px 그리드
  • 명도 대비 ≥ 4.5:1
  • 버튼 4 상태 (default/hover/pressed/disabled)
  • 빈 / 에러 상태 시안
  • 모션 (04-motion-spec) 준수
  • 마이크로카피 톤 (한국적)
  • 색맹 시 정보 손실 X (아이콘 / 텍스트 보조)
  • 모바일 터치 타겟 48×48
  • localization 길이 변화 대응

5-2. 새 컴포넌트 검증

(11-component-library 기준)

  • Figma 컴포넌트 등록
  • 모든 variant 시안
  • 모든 state 시안
  • Godot 코드 매핑 (개발자 합의)
  • 디자인 토큰 (색 / 폰트 / 사이즈) 사용

5-3. 새 화면 / 페이지 검증

  • 와이어프레임 (02-asset-spec 형식)
  • 데스크탑 + 모바일 (12-mobile-ui)
  • 한국어 + 영문 (10-localization)
  • 접근성 (09)
  • 사용성 테스트 시나리오

7. 접근성 테스트

7-1. 자체 검증 (디자이너)

  • Stark 플러그인 (Figma) — 명도 대비 자동
  • Color Blind 플러그인 — 색맹 시뮬레이션 (4종)
  • Chrome DevTools — Rendering tab Emulate vision deficiencies

7-2. 외부 베타 — 접근성 그룹

  • 색맹 사용자 2~3명 — 베타 키 우선 배포
  • 저시력 노인 1~2명 — 폰트 / 가독성
  • 청각 장애 사용자 1명 — 시각 대안

7-3. WCAG 검증

  • 모든 텍스트 ≥ 14px (보조 ≥ 12px)
  • 명도 대비 ≥ 4.5:1 (작은 텍스트)
  • 정보 색만으로 X (아이콘 / 텍스트)
  • 키보드 네비 (Phase 4)
  • 자막 / 시각 대안 (Sprint 3.13 완성)

8. 성능 테스트

8-1. Godot 디버거

  • FPS 모니터 (F12 또는 Game → Debug → Display FPS)

  • Profiler (느린 함수 식별)

8-2. 실기기 테스트

  • macOS Activity Monitor
  • Windows Task Manager
  • iOS Xcode Instruments (모바일)
  • Android Studio Profiler

8-3. 최적화 우선순위

1. 60 FPS 안정 (가장 중요)
2. 메모리 500 MB 이하
3. 로딩 < 3초
4. 절전 (모바일 / 노트북 배터리)

9. 베타 테스트 — 단계별

9-1. Closed Alpha (개발자 + 5명)

  • 인원: 5명 (핵심 베타)
  • 기간: 1주
  • 목표: Critical 버그 발견 + 사용성 첫 인사이트

9-2. Closed Beta (50~100명)

  • 인원: 50~100명 (페르소나 분포)
  • 기간: 2~4주
  • 목표: 디바이스 호환 + 밸런스 + 디자인 일관성

9-3. Open Beta (500명+)

  • 인원: 500명+ (Discord / Reddit / Steam)
  • 기간: 4~6주
  • 목표: 성능 / 글로벌 / 마케팅 검증

9-4. Release Candidate

  • 정식 출시 전 1주 freeze
  • Critical 버그만 hotfix
  • 인쇄 / 마케팅 자료 동시 발송

13. 디자이너 ↔ QA 협업

13-1. 디자이너 → QA 핸드오프

  • 새 시안 → QA 가 확인할 체크리스트 첨부

  • 사용자 피드백 → 디자인 개선 요청

14. 결론

좋은 QA = 사용자가 버그 / 시각 깨짐을 만나지 않도록.

K-Sutda 의 QA 핵심:
1. 자동 단위 테스트 (현재 3,197 passing — 유지)
2. 시각 회귀 (수동 + 미래 자동)
3. 사용성 테스트 (페르소나 P5 기준)
4. 디바이스 매트릭스 (macOS + Windows + 모바일)
5. 접근성 (색맹 + 저시력 + 청각)
6. 베타 단계별 (5명 → 100명 → 500명)

QA 는 출시 직전이 아닌, 매 sprint 의 일부.
